# AGENTS.md

まず [README.md](README.md) を読んでください。ここには README から読み取れないエージェント向けの指示だけを書きます。

- 補助ツールは Zig で書いてください (`tools/*.zig`)。Python やシェルスクリプトを持ち込まないでください。外部コマンドへの委譲は perf のように事実上代替が無い場合に限ります。
- mise のシェルフックが効かない非対話シェルでは、`mise run <名前>` の代わりに `mise x -- zig build <名前>` を使ってください。
- 性能について報告するときは `mise run bench` の中央値を根拠にしてください。単発実行の time を根拠にしないでください。
- データの分布に関わる設計判断 (ハッシュ・キャッシュ・分岐など) の前に、`datasets/` の生データで分布を実測してください。分布の仮定 (一様・ランダム) は実データに裏切られた実績があります。集計用のワンオフスクリプトは、結果を生データと突き合わせて検算してから信用してください。
- 提出後の判定は `mise run status` で確認できます (Web UI を見られないため。`-Did=<uuid>` で単体表示)。`status` が `infrastructure_error` (例: "worker restarted during benchmark") ならジャッジ側の問題なので、同じ提出物をそのまま再提出してください。
- 提出などの一回きりで結果が重要なコマンドの出力を `tail` / `head` で切らないでください。切ってよいのは内容が既知で冗長な出力 (ビルドログ等) だけです (提出エラーの本文を切り落として原因不明のまま進めた失敗例あり)。
- 提出 API は一時的に HTTP 5xx (`runner_unavailable` など) を返すことがあります。このとき提出はサーバー側に作られていないので、少し待ってそのまま再提出してください。
- `.env` の環境変数 (`$ONEBRC_ACCESS_KEY` など) は mise が載せるため、バックグラウンドシェルや素の `curl` には渡りません。必要なコマンドは `mise x -- sh -c '...'` で実行してください (キー無しで API を叩いて `authentication_required` になった失敗例あり)。
- `zig build` は C コンパイラの警告を表示しませんが、全ビルドに `-Werror` が付いているため、警告があるとビルドが失敗して可視化されます (`[[unsequenced]]` が黙って無視されていた実績への対策)。手動の警告チェックは不要です。`zig cc` を直接呼ぶ場合は、`-c -o /dev/null` や `-S` でもカレントディレクトリに `.o` / `.s` を吐くことがあるので、出力先を必ず作業用ディレクトリに明示し、実行後に `git status --short` で生成物が無いことを確認してください (リポジトリ直下を汚した失敗例が 2 回あり)。
- 実装で新しい分岐経路 (稀にしか通らないフォールバックなど) を作ったら、`tools/gen_edge.zig` にその経路を踏む行があるか確認し、無ければ追加してください (SWAR パースの 8 桁以上フォールバックが全データセットで未実行だった実績あり)。
