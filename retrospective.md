## TwoSquirrels の記録

- 言語は C。Zig を C コンパイラ + タスクランナーとして使い、musl 静的リンク・`-march=sapphirerapids` でビルドした。
- 提出は 33 回。初提出 `tight.c` が Public 106.655 秒。最速 Public は `27-robust` の 3.486 秒 (#31)、最終提出は同じ `27-robust` の 3.548 秒 (#33)。
- 最終解 `27-robust` の主な構成:
  - `mmap` + 8 スレッドで入力をバイト等分 (改行整列)。各スレッドがパースと集計を担当する**データ並列**。
  - 行パースは `SWAR` / `SIMD`。提出ビルドは `AVX-512` 有効で、区切り探索と数値変換は `SWAR`。
  - `16384` 枠の開番地ハッシュ表。指紋 (`first8` / `last8` / `mid8`) + 必要なら全バイト比較の厳密照合で、衝突安全。
  - 集計先の 3 段プリフェッチパイプライン (バッチ 16)。
  - 実行時に頻度を学習する ID リナンバリング (`renumber`)。頻出チャンネルを `stats` の先頭へ寄せる。
  - スレッド別のフル集計表を、最後に `merge` (共有状態なし)。
  - 出力は整数を自前整形し、平均値は `printf` 互換の丸めで、単一の `write`。
  - 資源・I/O 失敗への防御 (`renumber` の `malloc`、`pthread_create`、`write` の `EINTR` / 部分書き込み)。
- 運営 Q&A で確認した仮定: 整数は符号・先頭ゼロを含まない正準十進 / メッセージ 0 行のケースは無い / `channel_path` は `^[0-9A-Za-z_-]{1,20}(/[0-9A-Za-z_-]{1,20}){0,4}$`。
- バイト等分の負荷均衡を 100M で実測した。入力を 8 分割した各区間の行数の spread `(max−min)/max` は 0.017%、名前長合計の spread は 0.037%、全 8 区間の distinct チャンネルは 10000 で揃った。入力がチャンネル順に並んでいない保証から、どのバイト塊もほぼ同じ分布になる。感想戦で述べた「バイト数で切ったので偏っても各スレッド均等になる (はず)」を、この数字で裏取りした。

以降、`14-thp`・`batch16`・`24-populate` のような `番号-名前` 形式は、コンテスト中に試した自分たちの提出をそれぞれ指す (提出は全部で 33 回。詳細な設計と手元 A/B の記録は [`consideration.md`](consideration.md) にある)。

## 運営発表の事実

- 裏コンテストとして GPT-5.6 Sol Ultra (xHigh) に本番同等の環境で改善させたところ、Public 1.5 秒台 / Private 1.7 秒台に到達した。
- 運営の手元での言語別の傾向は C = C++ < Zig < C# < Rust < Go (左が速い)。TypeScript / Ruby は 1 桁以上遅く不向き。
- 運営が GPT-5.6 Sol に書かせた段階的な解説 ([guide](https://1brc.trap.games/guide)) は初歩的な内容で、最終的に 7.8 秒までの改善。
- pikachu さんは Public 2.426 秒に対し Private 7.362 秒と、Public / Private の差が大きく順位が下がった。
- メッセージ長の上限は仕様に明記されなかったが、運営は後日「本家 traQ の上限 10000 以下と仮定してもレギュレーション違反とはしなかった」と述べた (GPT-5.6 の解は念のため 10000 超にも対応している)。ただし公開データを実測すると `message_length` が 10000 を超える行は 100M 行中 17,528 件あり、最大値は 12000 だった。つまり「10000 以下」という仮定自体は厳密には誤りで、運営の発言は仮定が正しかったことの追認ではなく、外れていた仮定への温情だったことになる。

### 運営が挙げた攻略の要点

結果発表後、cp20 さんが解説していた内容。

- チャンネルをハッシュ化して文字列を持たないことで速くする。ただし衝突検知を併せて正しさを守る。
- チャンネルが 10000 種類に限られる保証から、**10000 種のユニークが出揃えば以降はハッシュ衝突が起きないことが保証される**。これが「飽和後は完全ハッシュ化できる」の公式な言語化で、上位解が独立に実装していた共通機構そのもの。
- 細かな技法として compiler intrinsics (アセンブリ相当)、LTO、PGO、CPU キャッシュ意識 (L1d ミス削減)、分岐予測ミス削減 (branchless)。
- 2 秒台では perf・flamegraph・吐かれたアセンブリを眺め、削って 10 ms 速く (遅く) を地道に繰り返す作業になる。
- 「仮定に書かれていないデータの特性を上手く使うのが上位の鍵。ただし、そうではないデータも正しく処理できる必要がある」。

## 1 位 zoi_dayo さんの提出を読んで

C++ で 2498 行。ほぼ全ての微決定を、40 個近いコンパイル時ノブ (`#define`) でパラメータ化した解。ソースは [`submissions/01-zoi_dayo.cpp`](submissions/01-zoi_dayo.cpp)。

### 骨格は我々と同型

- `mmap` + 8 スレッドで入力を**バイト等分** (改行整列)。
- 各スレッドがパースと集計を丸ごと担当する**データ並列**。パース段と集計段を別スレッドに分ける `producer-consumer` 型ではない。
- `16384` 級の開番地ハッシュ表。最後に `merge`。スレッドは CPU にピン留め。
- 使う SIMD は `AVX2` のみ。`AVX-512` は使っていない。

各ワーカーは二相で動く。まず厳密な辞書 (`find_or_insert`) でチャンネルを学習し、`10000` 種が出揃った瞬間に完全検索構造を構築・検証する。以降は分岐の少ない `hot loop` へ切り替える。構築や検証に失敗したら、厳密プローブのまま続ける (フォールバックあり)。

### 我々が消せなかった 3 つの毎行コストを、安全に消していた

**1. チャンネル ID を「プローブ + 比較」から「2 ロードの `XOR`」へ。**

`10000` 種が出揃った後に、`XOR` フィルタ (`peeling` ベースの `retrieval` 構造) を構築する。`hash` から左右 2 つのテーブルを引いて `XOR` するだけで `channel_id` が確定し、プローブもキー比較も要らない。構築後に全チャンネルで検証し、通らなければ厳密プローブへ戻すので衝突安全。

我々が `perf` で実測した最大のホットスポット (集計表の照合) を、厳密性を保ったまま丸ごと消している。我々は「厳密性を保つ限り毎行の照合は消せない」と結論づけたが、これは誤りだった。「チャンネル数の上限が `10000`」という保証を、キャッシュ配置ではなく**完全ハッシュの構築可能性**に使う道があった。

**2. タイムスタンプを数値化しない。**

時刻の用途は月だけ。だから数値化せず、先頭 5 桁の `ASCII` を `512` エントリの完全ハッシュ表へ写像して月を直接引く。月境界をまたぐバケットだけ、下位 5 桁を固定幅 `ASCII` のまま月初のカットオフと比較する。`SWAR` パースも `86400` の除算も無い。我々は `ts / 100` を `SWAR` で作って割っていた。

**3. 集計値が 16 バイト。**

`count` (30 bit) + `min` (17 bit) + `max` (17 bit) を 1 つの `uint64_t` に、`total_length` + `total_stamp` をもう 1 つの `uint64_t` に畳む。我々の `Stats` は 24 バイト。min / max の更新は cold な `noinline` 分岐 (`cmov` ではなく分岐。我々と同じ結論)。

集計の反映は、8 段の遅延リング (`SoA`) でプリフェッチと重ねる。我々の 3 段プリフェッチパイプラインを、段数 8 まで深めた一般化にあたる。

### 手法

- ほぼ全ての微決定 — `cmov` か分岐か、遅延段数、ハッシュ方式、メモリオペランド比較の可否など — を `#define` ノブにして、総当たりで掃引した痕跡がある。
- インライン `asm` は 3 箇所だけ。いずれも急所の `codegen` を固定するためのもの: `vpcmpeqb` (メモリオペランド)、`leaq` (メッセージポインタの加算)、`test` + `cmov` (2 桁 / 3 桁の分岐なし選択)。
- CPU ピン留め、`posix_fadvise(SEQUENTIAL)`、集計 `arena` だけ `THP`。

### 独立に同じ結論へ至っていた点

次は我々も独立に採用していた。

- `total_length` + `total_stamp` を 1 回の加算にパッキング。
- min / max は `cmov` ではなく分岐。
- データ並列のバイト分割。
- 平均値を `printf` 互換の丸めで自前整形。
- バッファリングした `write`。

### データ仮定の賭け

数値パースは `decode_uint_1_to_5_digits`。つまり **`message_length` ≤ 5 桁 (≤ 99999) を仮定**している。仕様本文には無い制約だが、本家 traQ の 1 メッセージあたりの文字数上限は 10,000 で、5 桁ならこれを大きく超えて覆える。

`channel_path` が「traQ と同じ制約」と確定した時点で、`message_length` も traQ の上限付近に収まると読むのが自然だった。ただし実際の公開データを調べると `message_length` の最大値は 12,000 で、17,528 / 100M 行が traQ の上限 (10,000) を超えていた。zoi_dayo さんが賭けたのは traQ の正確な上限そのものではなく、5 桁という余裕を持った上界で、これは 12,000 を難なく覆う。この緩め方が、結果的に実際の分布とも矛盾しなかった。外れていれば anko さんの「チャンネル名 ≤ 32 文字」と同じ失格リスクだったが、5 桁という桁数の仮定は traQ の厳密な数値上限より安全側に振っていた。我々は 10 桁まで安全側に取り、この仮定を取りにいかなかった。

### 差の総括

トップとの約 1.5 倍は、`ISA` でも別アルゴリズムでもなかった。我々と同じ骨格の上で、

- (a) 完全ハッシュ検索で毎行の照合を消し、
- (b) 時刻を数値化せず月を引き、
- (c) 集計を 16 バイトに畳み、
- (d) 全微決定を掃引し、
- (e) 急所を手彫りし、
- (f) 仕様外の仮定を賭けた。

その総和だった。特に (a) は、我々が開発中の考察 ([`consideration.md`](consideration.md)) で「厳密な照合を保つ限りこれ以上は削れない」と結論づけていた壁の、正しい越え方だった。

## 運営 GPT-5.6 Sol Ultra の提出を読んで

裏コンテストとして運営が GPT-5.6 Sol Ultra (xHigh) に改善させた解。Public 1.5 秒台 / Private 1.7 秒台で、優勝者より速い。C で 1498 行。公開されたのは全行に日本語の注釈と実測値が付いた解説版で、採用した技法だけでなく**棄却した代替案の数字**まで残っている。ソースは [`submissions/00-GPT-5.6-Sol-Ultra-xHigh.c`](submissions/00-GPT-5.6-Sol-Ultra-xHigh.c)。

### 最速でありながら、完全に安全

この解は zoi_dayo さんの賭けすらしていない。

- `message_length` ≥ 65535 は 32 bit の補助表 (`WideStats`) へ cold パスで退避する。5 桁仮定なし。
- 同一 32 bit ハッシュのキーは `UINT16_MAX` 番兵から全バイト比較へフォールバックする。衝突安全。
- チャンネルが 10,000 種すべて出揃った後に構築する MPH (最小完全ハッシュ表。詳細は後述) の構築失敗・10,000 種未満の入力・巨大ページ非成立の全てに、正確なフォールバックがある。
- 末尾の overread はガードページ (`mmap` でファイル長 + 1 ページを予約し、先頭だけ `MAP_FIXED` でファイルに置換) により構造的に安全。我々の `TAIL_SLACK` バッファ退避より綺麗な解法。

「速さは安全性との交換」という前提が偽だと示した。正当な全入力を処理できる解が、最速だった。

### 我々の実験結果と一致する注釈

注釈中の実測が、我々の実験と独立に一致する。

| GPT-5.6 の注釈 | 我々の実験 |
| --- | --- |
| 事前 fault は 0.3〜0.4 秒遅い | `24-populate` が本番 +9% |
| SMT 込み 8 スレッド > 4 スレッド | `20-fourthreads` が 1.38 倍悪化 |
| `MADV_SEQUENTIAL` は数 ms 以内 | `16-madvseq` が中立〜悪化 |
| 独自キャッシュ層は 0.8〜1.0 秒遅い | ホットセット案を分岐予測理由で棄却 |
| カウンタと極値の分離配列で 3.0 → 3.5 秒に悪化 | レイアウト変更 (`25-fuse`) の悪化と同根 |

### 核心の技法

- **`Stats` 16 バイト + 1 命令の極値更新**。`total_len` / `stamps` / `messages` (各 32 bit) + `min_len` / `inv_max_len` (各 16 bit)。最大値を `65535 - max` で反転保存すると、min と max を 1 命令の `VPMINUW` で同時更新できる。`Stats` 全体の更新が 128 bit SIMD の load → add → `ternarylogic` → min → store で完結し、分岐が無い。
- **飽和後の MPH (最小完全ハッシュ)**。10,000 種が出揃ったら、バケット別シード方式の二段表 (シード 4 KiB + ID 32 KiB) を構築する。以降は配列参照 2 回で ID が確定。zoi_dayo さんの `XOR` retrieval と同じ扉を別の鍵で開けている。飽和後の完全ハッシュ化は上位の共通解だった。
- **CAS による辞書共有**。最初に全種を発見したワーカーの辞書を `atomic` CAS で公開し、全ワーカーが同じ ID 空間へ正規化する。行ごとの経路にロックを置かず、ワーカー別のキー管理と併合時のキー比較を消す。
- **AVX-VNNI の内積でパース**。`_mm_dpbusd_epi32` 1 命令で、ASCII 数字列の整数化と `{message_length, stamp_count, 1, 0}` の増分ベクトル生成を同時に行う。区切りの分類は `CMP` + `CMOVE` ×2 (インライン asm) と `PEXT` で分岐を消す。
- **月引きが `>> 5` の表引き**。全ての月境界が 3,200 秒 (100 秒単位で 32) の倍数であることを使い、タイムスタンプ先頭 8 桁からシフト 1 回で月を引く。我々は 864 で除算していた。
- **`id * 3 + 1` 符号化**。`ChannelAgg` が 192 バイト = キャッシュライン 3 本であることを使い、集計先アドレスを少ない命令で計算する (約 40〜50 ms)。
- **レジスタ 5 本の遅延パイプライン**。定常ループでは 5 行前の集計をレジスタ上で遅延させ、集計先の prefetch とランダムアクセス待ちを重ねる (1.88 → 1.72 秒)。我々の `batch16` は同じ発想を配列往復で実装していた。
- **処理済み入力を 256 MiB ごとに `MADV_DONTNEED`**。ページテーブルの解体 (約 50 ms) を解析と並行させる。

### 我々の考察との痛い接点

`inv_max` の反転保存は、実は我々も開発中に発見していた技法だった。「min を `max(~len)` として持てば min/max が同種演算に揃い、SIMD 化時に 1 系統の命令で済む (補数トリック)」と `consideration.md` に書き、しかし「スカラーでは利得ゼロ」と評価して SIMD 集計そのものに着手せず、実装しないまま棚に置いていた。GPT-5.6 はその先の SIMD 集計本体まで作った。アイデアの有無ではなく、やり切りの差だった。

なぜやらなかったのかを掘ると、三層になる。

1. 補数トリックは「SIMD 集計に着手する時まで棚上げ」と条件付きで正しく評価していたが、その「SIMD 集計」を一度もスケジュールに載せなかった。
2. 載せなかったのは、集計の ALU コストを「タダ」と査定していたから。min / max を `cmov` 化したら逆に遅くなった実測 (分岐は温まればほぼ不成立 = 予測が当たる) と、`perf` 上の `update_stats` の時間がメモリ待ちに見えたことから、「集計の敵はレイテンシであって計算ではない」と結論し、プリフェッチ側だけを攻めた。
3. その査定は単品としては正しかった。GPT-5.6 の実測でも `VPMINUW` の極値更新は約 30 ms ≈ 1% で、これは我々が本番提出の A/B 比較で検出できる最小差 (同一バイナリを 2 回提出したときのブレが ±0.02 秒あった) とほぼ同じ大きさしかなく、単品で提出しても効果があったかどうか確認できない粒度だった。だが GPT-5.6 はこれを単品ではなく**束**で採っていた。極値が `uint16_t` ×2 に縮む → `Stats` が 24 → 16 バイト → 集計表 2.88 → 1.92 MB で 2 MiB 巨大ページ 1 枚に収まる → `ChannelAgg` がキャッシュライン 3 本ちょうど → `id * 3 + 1` のアドレス符号化まで連鎖する。我々は「`pmaxud` 1 系統で済む」という命令数の利得だけを値付けして棚に置いたが、実際の価値はレイアウト連鎖の起点であることにあった。

終盤の我々は「検出可能な (> 5%) 施策はもう無い」モードに入り、単品 1% に見える施策を自動的に落選させ続けた。**束で効く施策を単品の EV で査定してはいけない** — これがこの解から学ぶ、一番高くついた教訓だと思う。

### fork による計測窓からの脱出

起動直後に `fork` し、子が全作業と出力の `write` + `close` を終えてから pipe で親に通知、親は即 `_exit(0)` する。26 GB の mmap の PTE / VMA 解体は、孤児化した子側で親の死後に進む。コメントには「ジャッジが親プロセスだけを待つ実装なら、テールの解体コストが計測窓の外に出る」「プロセスグループ全体を待つ実装でも損はない」とあり、外れても無害な片側の賭けとして設計されている。

運営が公開したベンチマーカーのソース ([cp-20/1brc-trap](https://github.com/cp-20/1brc-trap)、`apps/runner/image/measure.mjs`) を見ると、この賭けは実際に成立していた。計測は Node の `child_process.spawn` で提出バイナリを直接起動し、**その子プロセスの `close` イベントで打ち切る**。`close` は spawn した直接の子プロセスだけを追跡し、その子がさらに `fork` して生んだ孫プロセスは対象に入らない。プロセスグループ全体を待つ処理も見当たらない。つまり Citrine さんの親プロセスが `_exit(0)` した瞬間に計測は止まり、孫プロセス側の mmap 解体は計測時間の外で進む。「プログラムを速くする」のではなく「計測される時間の定義を突く」というメタ層の一手で、我々も GPT-5.6 も攻めていない。`fork` 失敗時はインライン続行、子の異常死は親が終了コードを伝搬と、失敗系も揃えてある。

### 月判定はパースも表引きもしない

- Citrine さん: パースも表引きもしない。閾値はレジスタ常駐で、分岐もロードも無い。

### 単射 32 バイト鍵と、ドレイン時の厳密照合

名前 (≤ 32 B) を `0x80 + clen` でパディングした 32 バイトに畳む。ASCII は < `0x80` なのでパディングと実バイトが衝突せず、長さ込みで単射 — つまり 32 B ベクトルの等値が、そのまま厳密な同一性になる。我々が `first8` / `last8` / `mid8` / `middle_equal` と場合分けした照合が、ベクトル比較 1 命令に置き換わる。

厳密照合の置き場所も独特で、ホットパスは 18 bit 指紋の楽観一致で走り、遅延リング (`RINGD` = 4) の排出時に 32 B 完全比較で検証する。不一致 (構成可能なハッシュ衝突) は `recover_slow` が正確に回収する。「厳密な同一性判定」への答えが、これで 4 通り出揃った:

| 解 | 厳密同一性の担保方法 |
| --- | --- |
| TwoSquirrels | 毎行、指紋 + 必要なら全バイト比較 |
| zoi_dayo | 飽和後に `XOR` retrieval を構築して全キー検証 |
| GPT-5.6 | 飽和後に MPH を構築し、番兵から正確フォールバック |
| Citrine | 楽観実行 + ドレイン時の完全照合で検証 |

## 3 位 Ponjuice さんの提出を読んで

### 検証済みハッシュ専用表 + 適応ハッシュ選択

飽和 (10,000 種) を検出すると、既存のプローブ表を「ハッシュのみで引ける形式」へ書き換える。書き換え前に全チャンネルのプローブ列を検証し、曖昧さが 1 つでもあれば書き換えず文字列比較経路を続行する (衝突安全)。独自なのは、**候補ハッシュを 3 つ持ち、実データで無衝突と検証できた最も安いものを選ぶ**こと。

1. `suffix8` — 末尾 8 バイトだけの `CRC32` (最安)。
2. `aes2_folded` — 128 bit の fold 状態を `AESENC` 2 round で拡散。
3. `folded` — fold + `CRC32` (既定)。

ホットパスのハッシュコストがデータに適応する。厳密同一性の担保方法はこれで 5 通り出揃った: TwoSquirrels (毎行照合)、zoi_dayo さん (`XOR` retrieval)、GPT-5.6 (MPH)、Citrine さん (ドレイン時検証)、Ponjuice さん (検証済み書き換え + 適応ハッシュ)。

### Public / Private の +22% の説明候補

適応ハッシュがそのまま説明候補になる。Public データでは最安の `suffix8` が検証を通り 2.117 秒。Private では名前集合が変わり、`suffix8` の検証が通らず重い候補へ落ちた可能性が高い。正しさは検証で守られ、速度だけが段階的に下がる — 設計された劣化であって事故ではない (これは推測で、確定情報ではない)。

### 月判定は加算の桁上がり

ASCII 2 バイトを `'0'` の引き算もせずそのまま疎テーブル (4096 エントリ) の添字にする。`-0x3030` の補正はアドレスの displacement に畳む。テーブル値は `((month + 1) << 32) − boundary` で、bswap した下位桁を加算すると**桁上がりで月が確定**する。ロード 1 + 加算 1。これで月判定は 4 解 4 様になった:

| 解 | 月の求め方 |
| --- | --- |
| TwoSquirrels | `SWAR` で 8 桁パース → `/ 864` → 日番号表 |
| GPT-5.6 | SIMD 積和で 8 桁パース → `>> 5` 表 |
| Citrine | ASCII を 11 閾値と比較 → `popcount` |
| Ponjuice | 生 ASCII 添字 + 加算の桁上がり |

全員が「10 桁固定の ASCII」という同じ保証から、全員違う機構を導いている。

- チャンネル表 128 KiB + 月別集計 1.92 MB を 1 枚の 2 MiB 巨大ページ (`HotStorage`) に同居 — GPT-5.6 の辞書ページと同じ発想。

### インライン asm の徹底

上位 3 解で最も多くインライン asm を使っている。

- プローブループが `asm goto`。衝突経路をラベルへ追い出し、ホームヒットを `xor` + `cmp` + `ja` + `and` の 4 命令に固定。
- `SHLD` で lookup と月を 1 レジスタにパック。`MOVBE` 後の冗長 move を空の asm で抑止。`BLSMSK` の CF で「カンマ無し」を判定。

### 飽和後の同一性、6 つ目の方式

### `message_length` の u16 仮定

`parseMessageLength` は u16 を返し、4 桁以上のフォールバックで `@intCast` している。`message_length` ≥ 65,536 の行が来ると ReleaseFast では未定義動作になる。公開データの実測では最大値が 12000 で u16 の範囲 (≤ 65,535) に大きく収まるため、この仮定は安全だった。ただし「traQ の上限は 10000」という読みそのものは厳密には誤りで (実測 17,528 / 100M 行が 10000 を超える)、運営が後日「10000 以下と仮定してもレギュレーション違反としない」と述べたのは、仮定の正しさの追認ではなく温情だった。TwoSquirrels・GPT-5.6・Citrine さん・Ponjuice さんは仕様本文の保証 (グループ合計が u32 以内) だけを頼りに wide 退避で処理していた — traQ の上限を仮定しない、最も安全な選択だった。

comptime で月境界テーブルを生成し、`@compileError` でレイアウトを検査、ワーカーごとに arena、`errdefer` で spawn 失敗時の join、`std.Io` の新 API — 我々が `tools/` 群で使ったのと同じ Zig 0.16 世代の書き方で、エラー処理と最適化が同居している。運営の言語序列 (C = C++ < Zig) に対し、この一本は C 勢全員より速い。

4. 平均値は `rintl((long double)d * 100.0L)` で、printf 互換丸めではない (公開データでは一致を実測確認済みとコメントにある)。

Public 1B での完全一致を確認してから投げている。検証は手元のデータに対してだけ行い、未知データへの防御を全て落とした — 上位勢が積んだ安全層の正確な鏡像。

### 安全のコストが実測できた

- anko さん (無検証): Public 2.199 秒。
- 安全な上位勢: Private 2.35〜2.58 秒。

検証とフォールバックの層を全部積んでも失うのは 1 割前後で、層を捨てた代償は Private でゼロ点。上位 4 解と anko さんは「毎行の識別コストを消す」という同じ目的地に着いており、違いはフォールバックを持ったかどうかだけ。それが優勝と失格を分けた。Luke256 さんの「ハッシュ、衝突しないで」が指していたのは、まさにこの構図だった。

### 提出物としての疑問点

- 提出ソースに `mphf_data.h` が含まれておらず、埋め込まれたチャンネル集合そのものは提出物から読めない (バイナリには入っている)。ソースレビューの観点では不完全な提出。
- ルールは「データセット固有のハードコード」を明示的に禁止しており、公開データのチャンネル名一覧をバイナリに焼くのはこの条項に照らして際どい。ただし運営の失格理由は「≤ 32 仮定」と「衝突処理なし」の 2 点で、ハードコード条項は挙げられていない。

## 10 位 pikachu さんの提出を読んで

C++ で 2519 行。Public 2.426 秒 (3 位) に対し Private 7.362 秒 (10 位) と、全参加者で最も大きく順位を落とした。この転落は安全フォールバックが正しく作動した代償で、anko さんの裏返しにあたる。ソースは [`submissions/10-pikachu.cpp`](submissions/10-pikachu.cpp)。

### 本走前のサンプリングで速い解か遅い解かが一発で決まる

1. 本走の前に、入力の 8 箇所から各 262,144 行 (計約 200 万行) を並列サンプリングしてチャンネル辞書を作る。
2. 辞書から実行時 CHD 完全ハッシュを構築する (GPT-5.6 と同方式)。
3. `fast_safe()` — 辞書がちょうど 10,000 種、かつ 64 bit ハッシュが全て一意、かつ CHD 構築成功 — の場合だけ、ハッシュのみで引く高速パイプラインを選ぶ。
4. 一つでも欠ければ、全行を毎行キー比較つきの exact パイプラインで処理する。

この選択は `main` で一度きりで、ワーカー起動後に昇格する仕組みが無い。Public では条件が揃って 2.426 秒、Private では揃わず全行が低速経路で 7.362 秒。崖が二値になっている。

### Private で何が外れたか

候補は 3 つあり、どれでも同じ崖に落ちる。

- サンプリング漏れ: 200 万行のサンプルに最稀チャンネルが 1 種でも現れなければ `size < 10000`。出現頻度 10⁻⁶ 級のチャンネルは 200 万行で欠け得る。
- ハッシュ一意性の破れ: Private のチャンネル集合で 64 bit ハッシュが 1 組でも衝突すれば `hash_unique_` が落ちる。
- CHD 構築失敗。

GPT-5.6 (Private 1.7 秒) や zoi_dayo さん (2.35 秒) の飽和経路が Private でも効いていることから、Private にも 10,000 種は揃っていたと推測できる。本命はサンプリング漏れかハッシュ性質の破れ。いずれにせよ検証つきゲートが「信用できない」と正しく判定して、安全側に倒した。

### 他の上位勢との違いは発動タイミング

全員が「全 10,000 種を知った後はハッシュだけで引く」という同じ構造に至っているが、発動の仕方が違う。

| 解 | 高速化の発動タイミング |
| --- | --- |
| zoi_dayo・GPT-5.6・pirosiki | 本走中、各スレッドが飽和 (10,000 種目) に達した瞬間に昇格 |
| Citrine | 発動条件なし (楽観 + ドレイン検証は常時有効) |
| pikachu | 本走前、サンプリングの結果で一括決定。やり直しなし |
| anko | ビルド時に固定。フォールバックなし |

本走中に昇格する設計なら、サンプルの運と無関係に、実際に 10,000 種が出揃った時点で必ず速くなる。事前サンプリング方式は判定を早める代わりに、外れたときの保険が「全行低速」しかない。崖の落差が 3 倍もあるのは、開発を fast 側に集中して exact 側 (毎行 `resolve_exact` + キー比較) を磨いていないためと見える。7.362 秒は我々の 4.238 秒より遅い。

### それ以外の道具立ては上位勢と共通

- `BRC_*` のコンパイル時ノブが 40 個超。zoi_dayo さん・kyomu さんと同じ掃引スタイル。
- バッチ 32 のダブルバッファで parse → resolve → update の 3 段。我々の `batch16` 3 段パイプラインと同型。
- 月判定は AVX-512 閾値比較。集計は 16 バイト圧縮 + 溢れは spill 表。平均は `snprintf("%.2f")`。
- `message_length` ≥ 65,536 の wide 値も未知チャンネルも spill 表で受ける。正しさの網は全部張ってある。

### anko さんと並べると

賭けの結果が出揃う: 速くて壊れた (anko さん、失格)、速くて守った (上位勢)、遅くなったが守った (pikachu さん、10 位で得点)。pikachu さんの 7.362 秒は「フォールバックはタダではないが、ゼロ点よりはるかに高い」という実測値になった。Public / Private の二段計測は、まさにこの種の「見えるデータへの過適合」を炙り出すために機能した。

## 失格 yn さんの提出を読んで

C++ で 1176 行。Public 2.199 秒 (失格組の最速タイ)。失格理由は anko さんと同じ 2 点 (チャンネル名 ≤ 32 文字 / ハッシュ衝突処理なし) で、コードを読むとその 2 点が構造そのものだと分かる。ソースは [`submissions/99-yn.cpp`](submissions/99-yn.cpp)。

### 検証という概念が存在しない

- チャンネルは常に 32 バイト 1 ロード。`channel_length` は comma マスクの `tzcnt` で、カンマが無ければ黙って 32 を返す。33 文字以上は切り詰められ、フォールバックの走査ループ自体が無い。
- 指紋は masked 32 バイトを `AESENC` 2 段で 64 bit 化し、`lookup()` は指紋一致をそのまま同一チャンネルとして返す。プログラム全体のどこにも名前比較が無い。
- 他の全員が持っていた「discovery → 飽和 → 昇格」の二相構造すら無く、1 行目から最後までハッシュだけで同一性を判定する。

各解の同一性モデルを並べると、yn さんだけが正当化を持たない。

| 解 | ハッシュのみ照合をいつ・どう正当化したか |
| --- | --- |
| zoi_dayo・GPT-5.6・Ponjuice・pirosiki | 実行時に全 10,000 種で検証してから (フォールバック付き) |
| Citrine | 楽観実行 + ドレイン時に必ず完全照合 |
| anko | ビルド時に構築し、公開データで実測検証のみ |
| yn | 正当化なし。最初から仮定 |

### この賭けは Public ですら勝っていなかった

安全な zoi_dayo さん (Public 2.131 秒) と Ponjuice さん (2.117 秒) は、無検証の yn さん (2.199 秒) より速い。検証を飽和後の一回に押し込んでホットパスから消す設計が確立した結果、終盤には検証付きの解が無検証の解を追い越していた。この賭けは失格リスクを負って、Public の順位すら買えていなかった。

### 道具箱は一級

- 月判定は 6 つ目の独自方式。タイムスタンプの 2〜5 桁目を `PEXT` で BCD に詰め、64K の constexpr 表を引く。月境界を跨ぐバケットだけ `bswap` + ASCII 比較で解決する。二層構造は zoi_dayo さんと同じで、索引の作り方だけ違う。
- AVX-512 のマスクレジスタを定数置き場に使う。ワーカー開始時に `k1` / `k2` / `k4` へ定数マスクを `kmovd` し、ホットループでは `vpmaxuw ... {%k4}` と手書き asm で参照して blend 命令 1 個を消す。他の誰もやっていない。
- 32 バイトの重なりストア。`Stat` 20 バイトを 256 バイト境界の `ChannelStats` に padding した上で、更新は 32 バイトのロード / ストア (隣の stat には +0 が書かれるだけ)。月の添字計算がシフトになる。
- `ONEBRC_*` ノブ群: ハッシュ 6 種・数値パース 5 種・パイプライン深さ 1〜6 + リング版・256 エントリ前段キャッシュ・`QEMU_SAFE` (エミュレータ検証用)・プローブ計数、そしてホットループを `.p2align 6` や `nop` ×32 / 48 で整列させるノブまである。コード配置の nop 詰めを掃引対象にしたのはこの解だけ。
- `ONEBRC_INDEX_ROWS` — 改行位置を先に配列化してからパースする方式が ifdef で眠っている。我々が手元棄却した newline インデックス方式と同じもので、既定 off。
- 平均値は integer / binary64 分解の printf 互換丸め。失格した解も、丸めは正確に書いている。

### もう一つの地雷

ガードページが無い。`mmap` はファイルちょうどのサイズで、名前の 32 バイトロードは最終行付近で写像の外に届き得る (ファイル末尾がページ境界の 32 バイト以内に落ちた場合のみ SIGSEGV)。GPT-5.6・Ponjuice さん・anko さんが匿名予約 + `MAP_FIXED` で塞いだ穴が、ここでは開いたまま。発火が確率的なので表面化しなかっただけ。

### まとめ

kyomu さんと同じ `ONEBRC_*` ノブ文化で道具を極限まで磨きながら、同一性モデルだけは一番粗い仮定に置いた。そしてその仮定が買ったはずの速度は、安全な上位に既に追い越されていた。「検証はホットパスから消せる = 実質タダにできる」ことの、裏側からの証明になっている。

## 全部読み終えての感想

8 本 (GPT-5.6・上位 4 解・pikachu さん・失格 2 解) を読み終えて、コンテスト全体がどういう勝負だったのかがようやく見えた。

### 骨格は一つしかなかった

読んだ解は全員、ほぼ同じ骨格をしていた。`mmap`、入力のバイト等分によるデータ並列、スレッド別の開番地ハッシュ表、集計の遅延パイプライン (深さ 2〜8)、最後に merge。「何をするか」の層では、実質的に選択肢は一つに収束していた。個性が出たのはその一段下 — 同じ保証から何を彫り出すか — で、たとえば月判定は 6 解 6 様、集計 16 バイト化の詰め方も全員違った。だが呆れるのは、`slot_code = 3 × id` のアドレス符号化のような細部まで、上位 3 者が独立に同じ答えへ着いていることだ。この問題には最適解の形がかなり明確に存在して、全員がそこへ収斂していく途中だった。

### 勝負を分けたのは、たった一つの設計判断だった

チャンネル同一性の扱い、それだけでほぼ順位が決まっている。実行時間を計測できた 29 人 (最終順位 18 人 + 失格 11 人) のうち 11 人がここで失格し、上位はここを「安全に消す」7 通りの方法を発明し、我々は「消せない」と誤断して 1.5 倍差の 8 位に残った。

- 消せないと判断した我々: 毎行、指紋 + 全バイト比較。安全だが 1.5 倍遅い。
- 無検証で消した anko さん・yn さん: Public 最速級、Private でゼロ点。
- 検証してから消した上位勢: 最速かつ安全。

最大の技術的発見はこれだ。**検証は、消すのではなく「飽和後の一回」に押し込める**。10,000 種の上限保証は我々もキャッシュ配置 (`renumber`) に使ったが、正しい使い道は完全ハッシュの構築可能性だった。同じ保証を手にして、引き出した価値が違った。

そしてこの飽和後完全ハッシュ化を、運営は結果発表で「10000 種のユニークが出揃えば、以降はハッシュ衝突が起きないことが保証される」と公式の攻略として説明していた。我々が読解で独立に辿り着いた原理は、そもそも出題者が想定した正攻法だった。上位はそれを本走中・ビルド時・楽観 + 検証と、実装の置き場所を変えて全員が実装していた。

### 安全と速度のトレードオフは、上位には存在しなかった

読む前は「上位は危険な賭けで速いのだろう」と思っていた。逆だった。最速の GPT-5.6 が最も安全で、5 桁仮定すらしていない。anko さん・yn さんの無検証は、終盤には Public の順位すら買えていなかった (zoi_dayo さん・Ponjuice さんの安全な解の方が速い)。そして pikachu さんが「フォールバックの作動コスト = 3 倍、それでもゼロ点よりはるかに高い」を実測してくれた。賭けの 2 × 2 が全部実測で埋まったコンテストで、答えは「安全はやり方次第でタダになる。無検証は何も買わない」だった。我々が終盤に堅牢化へ舵を切った判断自体は正しかった — ただしそれを「速度と引き換え」だと思っていた点が間違っていた。

### 我々の計測は正しく、射程が足りなかった

悔しさの質を正確にしておきたい。GPT-5.6 の注釈と我々の実験記録は、事前 fault・SMT・`MADV_SEQUENTIAL`・独自キャッシュ・レイアウト分離の 5 項目で独立に一致した。つまり我々の測定と棄却は正しかった。負けたのは測定ではなく、仮説の射程だ。

1. 「厳密性を保つ限り毎行の照合は消せない」と証明抜きに結論した (実際は 7 通りの消し方があった)。
2. 補数トリックを自力で発見しながら、単品 1% と査定して棚に置いた (実際は 16 バイト化 → 巨大ページ → アドレス符号化と連鎖する束の起点だった)。
3. AVX-512 に幅を期待して賭けた (勝者たちは AVX2 で戦うか、レジスタ本数のために使っていた)。

どれも「測れば分かる」類ではなく、「思いつくか」の類。実験の手続きは正しくても、仮説空間が狭ければそこが上限になる。

### 情報も実力のうちだった

ここは公開されている運営のデータ生成コードで検算すると、単純な話ではなかった。「traQ と同じ制約」という一言を文字通り取って `message_length` ≤ 10,000 と仮定するのは、実は誤りだった。実データを見ると最大値は 12,000 で、10,000 を超える行が 100M 行中 17,528 件ある。zoi_dayo さんが賭けたのは traQ の正確な数値ではなく、**5 桁という余裕のある桁数**だった。これなら 12,000 を難なく覆いつつ、任意桁数の汎用パーサより速い。正確な数字を言い当てる必要はなく、実際の値より確実に緩い上界を選べば、それだけで安全かつ速かった。

我々はチャンネル制約の確認で止まり、`message_length` には桁数の仮定を一切置かず、生成された数値をそのまま汎用パースしていた。これは traQ の上限が実は誤りだったことを踏まえると結果的に正しい選択だが、5 桁という緩い上界を検討すらしなかった点は見劣りする。仕様の行間を読み切る必要はなく、安全に緩められる余地がないか一度検討していれば良かった。

cp20 さんは結果発表でこれを定式化していた —「仮定に書かれていないデータの特性を上手く使うのが上位の鍵。ただし、そうではないデータも正しく処理できる必要がある」。頻度の偏りは我々も `renumber` で使えたが、この「安全に緩められる数値上界」は検討にも上らなかった。情報を読む力とは、正確な数字を当てることではなく、どこまで緩めても安全かを見極めることだった。

### 開発様式も三者三様だった

上位のソースには開発の仕方まで写っていた。全微決定をノブにして掃引する様式 (zoi_dayo さん・kyomu さん・pikachu さん・yn さん)、版を重ねて削ぎ落とす様式 (Citrine さん、679 行)、移植可能な記述で言語処理系に任せる様式 (pirosiki さん、intrinsic ゼロで C 勢全員より速い)。どれでも上位に届いている。ただし共通するのは、手元で決着しない差分を大量に、系統的に試し切っていることだ。我々の 33 提出は同じ思想だったが、ノブ化して束で回す仕組みを持たなかったぶん、試行の密度で負けた。

### 締め

106.655 秒から 3.5 秒までの 30 倍は自力で削れた。そこから先の 1.5 倍は、読み終えた今なら全部具体的に言える — 飽和後の完全ハッシュ化、集計 16 バイトの束、時刻を数値化しない月判定、そして仕様の行間を読む質問。どれも次は最初から狙える。
